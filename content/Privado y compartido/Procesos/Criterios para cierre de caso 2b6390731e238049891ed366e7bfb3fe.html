🧭 Protocolo de Cierre de Casos – SAC TECH / Ingeniería

🎯 Objetivo

Estandarizar el flujo de gestión y cierre de casos, asegurando trazabilidad, consistencia, y alineación entre SAC, KAM, SAC TECH e Ingeniería.

🔵 1. Estado: NUEVO

Definición: Caso recién ingresado por SAC o KAM.

✔ Acciones de SAC TECH

  • Validar información mínima:
    • ID de ambiente
    • Descripción
    • Evidencia (si aplica)
    • Categoría correcta
  • Confirmar si corresponde a soporte operativo o análisis técnico.

➡ Movimiento a otro estado

  • A “En proceso” → cuando SAC TECH toma el caso para análisis activo.

🟡 2. Estado: EN PROCESO

Definición: Caso asignado y siendo trabajado por SAC TECH.

▶️ Cuándo mover a “En espera”

Falta información o acción del solicitante.

Ejemplos:

  • Falta evidencia
  • Falta pasos para replicar
  • Falta acceso
  • Falta contexto o autorización

Debe quedar registrado qué información falta.

▶️ Cuándo mover a “Escalado Ing”

El escalamiento ocurre antes de la priorización.

Usar cuando:

  • El caso corresponde a un bug/error real y replicable
  • SAC TECH ya reunió la documentación necesaria
  • No puede resolverse desde SAC TECH
  • Requiere análisis y/o fix de Ingeniería

Acciones:

  • Crear thread en Slack #new-seguimiento
  • Agregar pasos, evidencia, ID del ambiente y contexto

▶️ Cuándo mover a “Cerrado” (desde SAC TECH)

Se cierra directamente si SAC TECH resuelve completamente el caso.

Casos típicos:

  • Configuración mal aplicada
  • Error de uso
  • Ajuste operativo (reglas, permisos, configuraciones, impresoras, horarios, otros)
  • Resolución interna sin necesidad de Ingeniería

Sub-estados aplicables:

  • Resuelto
  • Sin respuesta
  • Rechazado – No replicable
  • Rechazado – No factible
  • Duplicado

Requisitos:

  • Evidencia de solución
  • Comentario final claro
  • Campos de resultado completados
  • Comunicación al solicitante (si corresponde)

🔴 3. Estado: ESCALADO ING

Definición: Caso enviado al equipo de Ingeniería.

✔ Lo que ocurre en esta etapa

  • Caso escalado sin necesidad de haber sido priorizado previamente
  • Se crea hilo en Slack #New-seguimiento https://toteat-espacio.slack.com/archives/C06EJHC9NHL
  • SAC TECH queda en seguimiento
  • Ingeniería aún no toma acción hasta tener prioridad

▶️ Priorización (posterior al escalamiento)

Responsables: Francisco Ramírez / Carlos Apablaza

Resultado de esta evaluación:

🔥 Prioridad P4–P5

→ Ingeniería toma el caso

→ Se crea ticket en GitHub

→ Se registra en:

  • Sheet “Solicitudes migración”
  • Notion “Entregas Solicitudes migración (Resueltos) ” una vez esté en producción.

💤 Prioridad P3 o menor → Sub-estado en Salesforce “Despriorizado”

→ No se trabajará en corto/mediano plazo

→ Puede ser reevaluado más adelante

→ Se registra en:

  • Sheet “Solicitudes migración”

→ No se cerrará el caso

Nota: El resultado de esta priorización será informado en el caso en Salesforce tras cada revisión.

🟢 4. Estado: CERRADO

Definición: Caso finalizado con su resultado correspondiente.

Sub-estados:

✔ Resuelto

Se solucionó correctamente (por SAC TECH o Ingeniería).

✔ Sin respuesta

El solicitante no respondió luego de varios intentos, se considerarán 7 días (reevaluar post análisis de casos).

✔ Rechazado – No replicable

El error no pudo ser reproducido por ingeniería, se dará el aviso al ejecutivo para levantar nuevamente en caso de volver a ocurrir el problema, para intentar extraer más información.

✔ Rechazado – No factible

No es técnicamente viable, queda fuera del alcance actual o no pertenece a la estrategia de negocio vigente.

✔ Duplicado

Existe otro caso principal asociado.

✔ Sin error encontrado

No se encontró registro del error a validar.

🧩 Criterios generales para cerrar un caso

Un caso solo debe cerrarse cuando:

  • Tiene diagnóstico final completo informado como publicación o comentario interno.
  • La solución está “En producción”.
  • El sub-estado refleja el desenlace real.
  • Se han completado los campos obligatorios para categorizar el caso.
  • Se etiqueta al solicitante en la publicación para asegurar la bajada de la resolución.
  • (Si fue escalado) Tiene trazabilidad en Slack / GitHub / Sheet / Notion

❌ Cuando no cerrar un caso

Un caso no debe cerrarse:

  • No tiene claridad en el diagnóstico o no está completamente definido.
  • La solución no es definitiva, Ingeniería aún está analizando o no está en producción el Fix.
  • No se ha validado la solución.
  • La persona que escaló el caso no ha sido informada.
  • El caso está despriorizado o es una mejora.
