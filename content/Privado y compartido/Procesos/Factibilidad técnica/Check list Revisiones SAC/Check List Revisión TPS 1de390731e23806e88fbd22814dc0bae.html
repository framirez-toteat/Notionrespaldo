1. Descripción

Determinar un paso a paso de acciones a validar para revisar TPS de manera correcta para un cliente que tiene TPS instalado previamente.

2. Alcance

Estas acciones deben ser realizadas por el equipo de soporte cuando se active una incidencia relacionada a impresión, y ya haya sido validado que el problema no corresponde a impresoras (cada impresora imprime su página de prueba correspondientemente). La acción de reiniciar el TPS debe realizarse única y exclusivamente cuando el problema esté asociado a impresión, cambio de parámetros en la impresión y cambios de estado (online/offline). Reiniciar el TPS no corrige errores relacionados a personal, maestro de productos, precios de lista etc.

3. Entrada

Cliente haciendo uso del chat de soporte y/o gestión solicitada interna.

4. Condiciones

Las personas que ejecuten el paso a paso deben:

  1. Validar que ambiente es el que está solicitando la revisión.
  1. Tener acceso a la herramienta Screenconnect.
  1. Validar que la conexión remota sea efectivamente al PC servidor del ambiente que realiza la consulta.

5. Procedimiento

  • Paso 1: Validación del ambiente

    Validar a qué ambiente corresponde la solicitud, si es que la solicitud es interna confirmar que el ejecutivo que la realiza envíe la información completa del ambiente a revisar, incluyendo nombre del ambiente y su respectivo ID (restaurante-local). Si la solicitud se realiza desde el chat de soporte Intercom, validar en la información del cliente en “User Data > See All”, donde se debe revisar el apartado “Ir” e “Il”, para el ID del local, y el apartado “Resto” para el nombre del ambiente desde el que el cliente está escribiendo.

  • Paso 2: Validación de conexión remota al servidor

    Realizar la conexión remota al PC servidor del ambiente solicitado a través de Screenconnect, se debe buscar al ambiente por su nombre en Support o Access, ya que dependiendo de como haya sido su implementación podría estar en uno u ambos, si está en ambos y/o el nombre repetido, se debe conectar a un equipo y confirmar con el cliente si el equipo al que se conectó es el correcto, en la parte superior saldrá un mensaje de “este equipo está siendo controlado por xxx”, si el cliente confirma eso, es por que está en el equipo correcto. Si detecta que hay más accesos para el ambiente y no corresponden al PC servidor, se debe marcar o eliminar para evitar confusiones posteriores.

  • Paso 3: Validación estado de TPS

    Lo primero que se debe revisar es el estado actual de TPS, para ello, se revisarán los íconos ocultos de Windows en la parte inferior derecha (o según corresponda según la configuración de Windows del cliente)

  • Paso 6: Archivo “Config” (Solo Legacy)

    Como última alternativa antes de reinstalar para corregir el TPS, podemos modificar el archivo “config” que es el que trae el token de este TPS, asociado al ambiente.

    Para esto se debe ir al ambiente del restaurante e ingresar a “Menú lateral > Configuración > Print Server y Api” y clickeamos sobre el lápiz de edición del TPS.

    Aquí veremos el token actual del TPS :

    Para realizar el cambio debemos dar al botón al costado de Token de Api, realizando esta acción se cambiará el Token en el cuadro de texto a la derecha. Una vez este sea diferente al que estaba en un comienzo, debemos grabar en la parte inferior.

    Una vez se haya guardado el cambio, entraremos nuevamente al API, ya con el Token nuevo, e iremos al ícono de copiar para sacar el texto dentro.

    Teniendo el texto copiado en el portapapeles, iremos a la carpeta raíz del TPS (Equipo > Disco C: > Toteat Print Server), donde crearemos un archivo de texto nuevo llamado “config”, copiaremos el texto dentro de él y grabaremos.

    Y ya cuando el archivo “config” esté guardado en la carpeta, procederemos a renombrar la carpeta “Data” en la misma carpeta, con un identificador correlativo (por ej: “Data1”)

    Y finalizaremos ejecutando nuevamente el TPS en el archivo Drone.exe

    Realizado todo esto, debiese ejecutarse de manera correcta el TPS.

    Nota: Todo el paso 6 es válido solamente para legacy, en new el token del API no cambia.

6. Salida

Realizando todos los pasos del proceso, el software TPS debiese ejecutarse de manera correcta, si no lo ejecuta aún, puede intentar reinstalar el TPS. Si aún así realizando todos los pasos no se abre correctamente, se debe escalar el caso con el coordinador o al área TECH para su validación, con los datos correspondientes.

7. Errores comunes

  • Errores más comunes del paso a paso en la revisión del TPS

    1. Fatal error detected: Este error se presenta una vez instalado el TPS, justo antes de ejecutarse correctamente. Este error se puede dar por alguna falla en el servidor, para casos excepcionales. Regularmente este error está referido a algún problema en el proceso de implementación (instalación) del TPS, para solventarlo, debe desinstalar el TPS, borrar las descargas del TPS bajadas previamente (en las descargas de windows), en TRM borrar el API del TPS, Refrescar, bajar el TPS y reinstalar todo de 0 y limpio en el sistema.

    1. Failed to Execute Script pyi_rth_win32comgenpy: Este error está provocado debido a una librería faltante en el Windows instalado “pywin32”, para resolverlo se debe actualizar el Windows o al menos las librerías correspondientes a pythoncom o win32com . Este proceso es muy lento y demorado, una opción es utilizar un 2do PC como servidor y ya no el que está dando el problema, e incluso en algunas ocasiones es necesario formatear el PC si es que aún instalando las librerías no funciona.

8. Punto de control

Para validar que el TPS se ejecuto de manera correcta, debe estar el ícono verde del TPS en la parte inferior derecha y los 9 servicios drone.exe corriendo de manera correcta en el administrador de tareas.
